Subscribe to BOTH tables — cascade-deleted pair_member rows do not fire events (Critical Finding #1). Derive `const swimmerCount = pairs.reduce((s, p) => s + p.members.length, 0)` and `const pairCount = pairs.length`. Render `<ConnectionBanner status={connectionStatus} onRefresh={refreshPairs} />` then `<SessionBoard poolName={poolName} swimmerCount={swimmerCount} pairCount={pairCount} sessionId={sessionId} poolId={poolId} pairs={pairs} onPairMutated={refreshPairs} isRefreshing={isRefreshing} />` inside a `min-h-screen bg-white` wrapper. |-----------|----------|-----------|-------------|-----------------|
| T-05-03 | Information Disclosure | Realtime subscription on anon key | accept | RLS skipped for v1 (STATE.md); subscription only signals that a change occurred — the actual pair data is re-fetched through the auth-gated Route Handler (Plan 05-03) |
| T-05-08 | Denial of Service | Channel leak / event burst | mitigate | useEffect cleanup uses `supabase.removeChannel` (prevents Strict-Mode TooManyChannels leak); 150ms debounce collapses the trio INSERT burst into one refetch (RESEARCH Pitfalls 2 + 3) |
| T-05-02 | Tampering | sessionId used in channel filter + fetch URL | mitigate | sessionId comes from server-provided props (page.tsx), not user input; the fetch target is the auth-gated Route Handler which re-validates and parameterizes the query |
