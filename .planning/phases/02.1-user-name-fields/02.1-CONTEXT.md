# Phase 2.1: User Name Fields - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

> **Execution note:** User confirmed this is a simple phase ‚Äî skip research, plan check, and verification. Go straight to planning and execution.

<domain>
## Phase Boundary

Admin can enter a counselor's real first and last name when creating an account. The name is stored on the user record and displayed in the User Management table alongside the username ‚Äî making it easy to identify accounts without relying on login handles alone.

**Scope anchor:** This phase touches only the user schema, the CreateUserModal form, the UserTable display, the server action, and a one-time backfill script for the existing seed admin. No other screens or workflows are affected.

**Success criteria:**
1. Admin can enter first name and last name when creating a new user account
2. The User Management table shows a Name column (first + last) for every user
3. The seed admin account has a name after the migration and backfill run

</domain>

<decisions>
## Implementation Decisions

### Schema
- **D-01:** Add two new nullable columns to the Drizzle `user` table: `firstName text("first_name")` and `lastName text("last_name")`. Nullable in the DB (safe migration, no default needed). Required at the form level only.
- **D-02:** Run `npx drizzle-kit push` after schema update. Same pattern as the Phase 2 admin plugin migration.

### Form ‚Äî CreateUserModal
- **D-03:** Add First Name and Last Name fields to the CreateUserModal, both required (`required` attribute on the inputs). Field order: First Name ‚Üí Last Name ‚Üí Username ‚Üí Password ‚Üí Role.
- **D-04:** `createUserAction` reads `firstName` and `lastName` from FormData and writes them to the DB via `data: { username, firstName, lastName }` in the `auth.api.createUser` body. The existing `name` field continues to be set to `username` (Better Auth's built-in ‚Äî leave unchanged to avoid side effects).

### Table ‚Äî UserTable
- **D-05:** Add a "Name" column as the first column. Table becomes: **Name | Username | Role | Created | Actions**. Name cell shows `${firstName} ${lastName}`. When both are null (legacy rows that haven't been backfilled), fall back to a dash `‚Äî`.
- **D-06:** The `User` type in UserTable gains `firstName: string | null` and `lastName: string | null`. The `page.tsx` query result cast is updated to include these fields.

### Backfill
- **D-07:** Create `scripts/backfill-admin-name.ts` ‚Äî a one-time script that sets firstName and lastName for the seed admin user (e.g. `firstName: "Camp"`, `lastName: "Admin"`). Same pattern as `scripts/backfill-admin-role.ts`. The planner can choose a sensible placeholder name or leave it configurable via a script constant.

### Claude's Discretion
- Exact placeholder name for the seed admin backfill ‚Äî "Camp Admin" or similar is fine; planner decides.
- Tailwind styling for the new First Name / Last Name fields ‚Äî match existing field style in the modal exactly (same classes as Username field).
- Whether to add `autoComplete="given-name"` / `autoComplete="family-name"` to the new inputs ‚Äî standard accessibility practice, apply it.

</decisions>

<specifics>
## Specific Ideas

- Name column is the leftmost column in the table ‚Äî admin sees the human name first, then the login handle.
- Fallback for nameless legacy rows: dash `‚Äî` in the Name cell (not an empty string, not "Unknown").
- No email field changes ‚Äî email remains the auto-derived `${username}@buddyboard.local` dummy.

</specifics>
