# Phase 2.1: User Name Fields — Discussion Log

**Date:** 2026-06-28
**Participants:** User + Claude

---

## Areas Discussed

### 1. Name Storage Strategy

**Question:** How should first and last name be stored on the user record?

**Options presented:**
- Two new columns (firstName + lastName) — structured, sortable, requires DB migration
- Reuse existing `name` field — no migration, but fragile string-splitting

**Decision:** Two new columns. Worth the small migration cost for clean, structured data.

---

### 2. Table Display

**Question:** How should name appear in the User Management table?

**Options presented:**
- Add a Name column (Name | Username | Role | Created | Actions)
- Replace Username column with Name
- Show combined in Username cell

**Decision:** Add a new Name column as the leftmost column. Admin sees real name and login handle side by side.

---

### 3. Required Fields

**Question:** Are first and last name required when creating a new user?

**Options presented:**
- Both required (form validation, DB nullable)
- Optional — admin can skip

**Decision:** Both required at the form level. DB columns are nullable to allow safe migration of existing rows.

---

### 4. Backfill for Existing Users

**Question:** How to handle the existing seed admin user who has no first/last name?

**Options presented:**
- Nullable columns + one-time backfill script (mirrors Phase 2 role backfill pattern)
- Nullable columns, leave existing users blank
- Empty string defaults

**Decision:** Nullable columns + backfill script for seed admin. Same pattern as `scripts/backfill-admin-role.ts`.
