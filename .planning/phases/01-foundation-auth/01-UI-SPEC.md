## Spacing Scale

Declared values (multiples of 4 — Tailwind default scale):

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `p-1` / `gap-1` | Icon gaps, inline padding |
| sm | 8px | `p-2` / `gap-2` | Compact element spacing |
| md | 16px | `p-4` / `gap-4` | Default element spacing, form field gaps |
| lg | 24px | `p-6` / `gap-6` | Card padding, section padding |
| xl | 32px | `p-8` / `gap-8` | Login card padding, layout gaps |
| 2xl | 48px | `p-12` / `gap-12` | Page-level vertical centering adjustment |
| 3xl | 64px | `p-16` / `gap-16` | Page-level spacing (used sparingly) |

**Exceptions:**
- Minimum touch target height: 44px for all interactive elements (inputs, buttons) — Apple HIG iPad requirement (source: iPad-first context). Implement with `min-h-[44px]` on inputs and buttons.
- Login button width: `w-full` within the form — full width of the form container.
- Pool selection buttons: `w-full` within the pool list container — large tap area.

## Component Contract

These are the only two screens and their required elements. No other UI exists in this phase.

### Screen 1: Login Page (`/login`)

**Layout:** Full viewport height, centered card, max-width 400px, horizontally centered.

```
Page background: bg-white, h-screen, flex items-center justify-center
  └── Form card: bg-slate-50, rounded-lg, p-8, w-full max-w-sm, shadow-sm
        ├── Heading: "BuddyBoard" — text-3xl font-semibold text-slate-900, text-center, mb-8
        ├── Form: flex flex-col gap-4
        │     ├── Username field group
        │     │     ├── Label: "Username" — text-base font-semibold text-slate-900
        │     │     └── Input: type="text", name="username", required
        │     │           min-h-[44px] w-full border border-slate-300 rounded-md px-3
        │     │           text-base text-slate-900 placeholder:text-slate-500
        │     │           focus:border-blue-500 focus:ring-1 focus:ring-blue-500 focus:outline-none
        │     ├── Password field group
        │     │     ├── Label: "Password" — text-base font-semibold text-slate-900
        │     │     └── Input: type="password", name="password", required
        │     │           (same classes as username input)
        │     ├── Error message (conditional, shown when login fails)
        │     │     └── <p>: "Invalid username or password"
        │     │           text-base text-red-600, role="alert"
        │     └── Submit button: "Log in"
        │           type="submit", min-h-[44px] w-full
        │           bg-blue-600 hover:bg-blue-700 text-white
        │           text-base font-semibold rounded-md
        │           transition-colors duration-150
        └── (nothing else — no "Forgot password", no "Remember me", no social login)
```

**Interaction states:**
- Default: form with no error message visible
- Submitting: button should show disabled state — `disabled:opacity-50 disabled:cursor-not-allowed` — while the auth request is in flight
- Error: error paragraph appears below the password field, above the submit button; username and password fields retain their values (do not clear on error)
- Success: JavaScript redirects to `/pools` (no visible transition state needed for MVP)

**Validation:** Both fields are `required`. Browser native validation is sufficient for MVP. No custom inline validation messages.

### Screen 2: Pool Selection (`/pools`)

**Layout:** Full viewport height, centered content, max-width 400px.

```
Page background: bg-white, min-h-screen, flex items-center justify-center
  └── Container: bg-slate-50, rounded-lg, p-8, w-full max-w-sm, shadow-sm
        ├── Heading: "Select a Pool" — text-xl font-semibold text-slate-900, text-center, mb-6
        └── Pool list: flex flex-col gap-3
              ├── Button "Main Pool"
              ├── Button "Lap Pool"
              └── Button "Kiddie Pool"
                    Each button: min-h-[44px] w-full
                    bg-blue-600 hover:bg-blue-700 text-white
                    text-base font-semibold rounded-md
                    transition-colors duration-150
```

**Placeholder note:** These buttons do not navigate anywhere in Phase 1. They are present to prove auth routing works and to provide real click targets for manual testing. Phase 4 will replace this screen with real session-start logic.

## Interaction & Accessibility Contract

| Requirement | Specification |
|-------------|---------------|
| Minimum touch target | 44px height on all inputs and buttons (Apple HIG iPad) |
| Focus ring | `focus:ring-1 focus:ring-blue-500 focus:outline-none` on all inputs |
| Error announcement | Error paragraph uses `role="alert"` for screen reader announcement |
| Color contrast | white text (#FFFFFF) on blue-600 (#2563EB) meets WCAG AA (4.7:1 ratio) |
| Color contrast | red-600 (#DC2626) on white (#FFFFFF) meets WCAG AA (4.5:1 ratio) |
| Color contrast | slate-900 (#0F172A) on slate-50 (#F8FAFC) meets WCAG AA (17.6:1 ratio) |
| Input autocomplete | `autocomplete="username"` and `autocomplete="current-password"` on login inputs |
| Form semantics | `<form>` element with `<label>` elements explicitly associated to inputs via `htmlFor`/`id` |
| iPad orientation | Layout works in both portrait and landscape on iPad (max-w-sm card handles both) |
