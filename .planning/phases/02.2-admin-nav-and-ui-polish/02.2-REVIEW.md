# Phase 02.2: Code Review Report

**Reviewed:** 2026-06-28
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed six source files and the actions module they reference (included to resolve
cross-file questions raised during reading). The phase correctly adds bidirectional nav
links, role-based post-login redirect, and icon-button replacements for user-table
actions. The authorization guard on the admin page is server-side and sound.

Two critical issues were found: the password reset server action enforces no
server-side length policy (client-only validation is trivially bypassed), and the login
form's error handling will silently fail on network-level errors. Four warnings cover
a double-redirect for unauthenticated users, two accessibility gaps, and missing success
feedback for the reset-password flow.

## Critical Issues

### CR-01: Server action sets passwords with no length validation

**File:** `src/app/(admin)/admin/users/actions.ts:47-56`

**Issue:** `setUserPasswordAction` calls `auth.api.setUserPassword` without checking the
`newPassword` argument at all. The 8-character minimum exists only in the client
component (`ResetPasswordForm.tsx` lines 20-23 and the `minLength` HTML attribute).
Next.js server actions are HTTP endpoints; they can be called directly — bypassing both
React form validation and the `minLength` attribute — to set a one-character or empty
password for any user. Because the function already requires admin authentication (`requireAdmin()`),
the attacker must be an admin, but the password policy should still be enforced at the
trust boundary.

**Fix:**
```ts
export async function setUserPasswordAction(
  userId: string,
  newPassword: string,
) {
  await requireAdmin();

  if (!newPassword || newPassword.length < 8) {
    throw new Error("Password must be at least 8 characters.");
  }

  await auth.api.setUserPassword({
    body: { userId, newPassword },
    headers: await headers(),
  });
}
```

### CR-02: Login form has no `catch` — network errors crash the page

**File:** `src/app/(auth)/login/page.tsx:19-34`

**Issue:** `handleSubmit` uses `try { ... } finally { ... }` with no `catch` block.
When `authClient.signIn.username()` encounters a network-level error (offline, DNS
failure, server timeout), the exception propagates out of the async event handler as
an unhandled promise rejection. In React 19 this surfaces to the nearest error boundary;
without an explicit boundary on the login page the entire page tree can unmount, leaving
the user on a blank screen or generic error UI with no actionable message.

The `better-auth` client normally returns `{ data, error }` tuples rather than throwing,
but relying on that undocumented behaviour makes the code fragile across library updates.

**Fix:**
```ts
try {
  const { data, error: signInError } = await authClient.signIn.username({
    username: formData.get("username") as string,
    password: formData.get("password") as string,
  });

  if (signInError) {
    setError("Invalid username or password");
  } else {
    if (data?.user?.role === "admin") {
      router.push("/admin/users");
    } else {
      router.push("/pools");
    }
  }
} catch {
  setError("An unexpected error occurred. Please try again.");
} finally {
  setLoading(false);
}
```

## Warnings

### WR-01: Optional chaining on `data?.user?.role` silently misroutes on null data

**File:** `src/app/(auth)/login/page.tsx:28`

**Issue:** If `authClient.signIn.username()` returns `{ data: null, error: null }` (a
degenerate success response), the optional chain `data?.user?.role` evaluates to
`undefined`, which is not `"admin"`, so the user is routed to `/pools`. `/pools` then
checks auth and redirects to `/login`, resulting in a confusing loop and no error
message. The fix from CR-02 applies; additionally, add an explicit guard:

**Fix:**
```ts
if (signInError || !data?.user) {
  setError("Invalid username or password");
} else {
  router.push(data.user.role === "admin" ? "/admin/users" : "/pools");
}
```

### WR-02: Unauthenticated users hit `/pools` before being redirected to `/login`

**File:** `src/app/(admin)/admin/users/page.tsx:13`

**Issue:** `redirect("/pools")` is used for both "not authenticated" and "not admin"
cases. A user who is not logged in at all is sent to `/pools`, which then calls
`redirect("/login")` — a two-hop redirect that adds latency and obscures intent.

**Fix:** Distinguish the two conditions:
```ts
const session = await auth.api.getSession({ headers: await headers() });
if (!session) redirect("/login");
if (session.user.role !== "admin") redirect("/pools");
```

### WR-03: Password reset input has no `<label>` element

**File:** `src/app/(admin)/admin/users/components/ResetPasswordForm.tsx:47-57`

**Issue:** The expanded reset form renders an `<input id="newPassword">` with no
corresponding `<label htmlFor="newPassword">`. The placeholder text
`"New password (min. 8 characters)"` disappears as soon as the user starts typing and
is not announced by screen readers as a label. This is a WCAG 2.1 SC 1.3.1 (Info and
Relationships) failure.

**Fix:**
```tsx
<label
  htmlFor="newPassword"
  className="block text-sm font-medium text-slate-700 mb-1"
>
  New password
</label>
<input
  id="newPassword"
  name="newPassword"
  type="password"
  required
  placeholder="Min. 8 characters"
  minLength={8}
  autoComplete="new-password"
  className="min-h-[44px] w-full border border-slate-300 rounded-md px-3 text-base text-slate-900 placeholder:text-slate-500 focus:border-blue-500 focus:ring-1 focus:ring-blue-500 focus:outline-none"
/>
```

### WR-04: No success feedback after password reset

**File:** `src/app/(admin)/admin/users/components/ResetPasswordForm.tsx:26-28`

**Issue:** On a successful `setUserPasswordAction` call the form silently closes
(`setOpen(false)`). There is no toast, inline confirmation, or any other signal that
the password was actually changed. An admin has no way to distinguish "it worked" from
"the form closed on its own". This can lead to repeated reset attempts and distrust in
the UI.

**Fix:** Set a transient success message before closing, or use a toast:
```ts
await setUserPasswordAction(userId, newPassword);
setOpen(false);
setError(null);
// Option A — inline confirmation via parent callback prop
// Option B — a short-lived state flag:
// setSuccessMessage("Password updated.");
```
The exact mechanism (callback prop, toast library, or parent-managed state) should match
whatever success-notification pattern is adopted project-wide.

## Info

### IN-01: Icon-only buttons use `title` instead of `aria-label`

**File:** `src/app/(admin)/admin/users/components/ResetPasswordForm.tsx:38-44`
**File:** `src/app/(admin)/admin/users/components/DeleteConfirmDialog.tsx:31-36`

**Issue:** Both round icon buttons carry a `title` attribute for their accessible name.
`title` is inconsistently exposed by screen readers and is not announced at all by some
combinations of browser + AT. `aria-label` is the WCAG-recommended mechanism for
naming icon-only controls.

**Fix:** Replace `title` with `aria-label` (keeping `title` for the visible tooltip is
fine):
```tsx
<button
  onClick={() => setOpen(!open)}
  aria-label="Reset password"
  title="Reset password"
  className="w-10 h-10 rounded-full ..."
>
  <Key size={18} />
</button>
```

### IN-02: Pool selection buttons have no click handlers

**File:** `src/app/(protected)/pools/page.tsx:21-27`

**Issue:** The three pool buttons render correctly but have no `onClick` handler —
clicking them does nothing. This is likely an intentional placeholder (pools feature is
not yet implemented), but it means the page is non-functional from an end-user
perspective and should be tracked.

**Fix:** Track in backlog and add a `disabled` attribute or visual "coming soon" state
until the feature is implemented, so the UI does not mislead users.

### IN-03: `toLocaleDateString` hardcodes `"en-US"` locale

**File:** `src/app/(admin)/admin/users/page.tsx:30`

**Issue:** `u.createdAt.toLocaleDateString("en-US", { ... })` hardcodes the locale on
the server. If the app is ever used in a non-US locale the date format will remain
US-centric regardless of browser/OS settings.

**Fix:** Consider using `undefined` to inherit the server's system locale, or pass the
locale through a configuration constant so it is easy to change in one place:
```ts
createdAt: u.createdAt.toLocaleDateString(undefined, {
  month: "short",
  day: "numeric",
  year: "numeric",
}),
```
